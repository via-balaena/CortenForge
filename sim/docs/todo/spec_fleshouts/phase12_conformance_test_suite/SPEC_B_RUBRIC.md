# Layer C Trajectory Comparison — Spec Quality Rubric

Grades the Spec B spec on 10 criteria. Target: A+ on every criterion
before implementation begins. A+ means "an implementer could build this
without asking a single clarifying question — and the result would produce
numerically identical trajectory output to MuJoCo."

**MuJoCo conformance is the cardinal goal.** Every criterion in this rubric
ultimately serves conformance. P1 (MuJoCo Reference Fidelity) is the most
important criterion — grade it first and hardest. A spec that scores A+ on
all other criteria but has P1 wrong is worse than useless: it would produce
a clean, well-tested implementation of the wrong behavior.

Grade scale: A+ (exemplary) / A (solid) / B (gaps) / C (insufficient).
Anything below B does not ship.

## Empirical Ground Truth

### MuJoCo behavioral verification

**MuJoCo version:** 3.4.0
**Reference data:** Generated by `gen_conformance_reference.py` (Session 4),
checked into `sim/L0/tests/assets/golden/conformance/reference/`.

### EGT-1: MuJoCo `mj_step()` pipeline ordering

MuJoCo's `mj_step()` in `engine_forward.c` (MuJoCo 3.4.0):

```c
void mj_step(const mjModel* m, mjData* d) {
    mj_markStack(d);
    mj_forward(m, d);    // Full forward pass: FK → CRBA → RNE → passive → collision → constraint → actuation → sensors
    mj_checkAcc(m, d);   // NaN/divergence check on qacc
    mj_advance(m, d);    // Integration: qpos ← qpos + dt*f(qvel, qacc); qvel ← qvel + dt*qacc
    mj_freeStack(d);
}
```

**Capture timing:** `gen_conformance_reference.py` captures `data.qpos`,
`data.qvel`, `data.qacc` AFTER `mj_step()` returns. This means:
- `qpos[step]` = integrated position (post-advance state)
- `qvel[step]` = integrated velocity (post-advance state)
- `qacc[step]` = acceleration computed by forward pass BEFORE advance
  (MuJoCo saves qacc from forward, then advance updates qpos/qvel but
  leaves qacc unchanged)

CortenForge's `Data::step()` at `forward/mod.rs:212` follows the same
pattern: `forward() → check_acc() → integrate()`. After `step()` returns,
`data.qpos`, `data.qvel`, `data.qacc` have the same semantics as MuJoCo.

### EGT-2: Integrator selection

All canonical models use `<option timestep="0.002"/>` without specifying
an integrator, so MuJoCo defaults to **Euler** (`mjINT_EULER`).
CortenForge also defaults to Euler when no integrator is specified in MJCF.

Euler integration in MuJoCo (`engine_forward.c:mj_Euler()`):
```c
// Semi-implicit Euler: update velocity first, then position
qvel += dt * qacc;
mj_integratePos(m, d->qpos, d->qvel, dt);  // handles quaternion integration for free joints
```

CortenForge's Euler integration at `forward/mod.rs:236` calls
`self.integrate(model)` which matches this semi-implicit pattern.

**Failing stages and root causes:**

| Root cause | Layer B tests affected | Trajectory impact |
|-----------|----------------------|-------------------|
| xipos not computed | 23 (FK:7, CRBA:7, RNE:7, sensor:2) | **CRITICAL** — wrong CRBA/RNE means wrong qacc at EVERY step. 7 of 8 models diverge from step 1. |
| Contact position convention | 2 (collision) | **MEDIUM** — affects contact_scenario trajectory via wrong contact force magnitude. |
| Constraint Jacobian assembly | 3 (constraint) | **HIGH** — wrong efc_J means wrong constraint forces for contact_scenario, equality_model, composite_model. |

**Two distinct divergence patterns for trajectory tests:**

1. **xipos-affected models** (pendulum, double_pendulum, actuated_system,
   tendon_model, sensor_model, equality_model, composite_model): CRBA and
   RNE are wrong → qacc is wrong from step 1 → trajectory diverges
   immediately. Even though passive, actuation, and tendon stages pass,
   the mass matrix (CRBA) and bias forces (RNE) dominate qacc. These
   models will show large divergence at step 1.

2. **contact_scenario** (unique): FK, CRBA, and RNE all pass. Passive
   passes. Only collision and constraint fail. This model's dynamics
   pipeline is correct — trajectory divergence comes from wrong constraint
   forces. It may match for several steps before contact-related divergence
   becomes significant, especially if the sphere is in free flight early
   in the trajectory. Divergence onset depends on when constraint forces
   first contribute to qacc.

**Expected trajectory test outcomes:**
- pendulum: `#[ignore]` — qacc wrong from step 1 (CRBA/RNE xipos cascade)
- double_pendulum: `#[ignore]` — qacc wrong from step 1 (CRBA/RNE xipos)
- contact_scenario: `#[ignore]` — constraint forces wrong (may match early steps)
- actuated_system: `#[ignore]` — qacc wrong from step 1 (CRBA/RNE xipos)
- tendon_model: `#[ignore]` — qacc wrong from step 1 (CRBA/RNE xipos)
- sensor_model: `#[ignore]` — qacc wrong from step 1 (CRBA/RNE xipos)
- equality_model: `#[ignore]` — qacc wrong from step 1 + constraint wrong
- composite_model: `#[ignore]` — qacc wrong from step 1 + constraint wrong

**All 8 trajectory tests will likely be `#[ignore]`d.** This is expected
and correct — the tests encode the MuJoCo ground truth and track progress
toward conformance. The tests must NOT be weakened.

**Diagnostic implication:** The `#[ignore]` tracking comment should
distinguish "diverges at step 1" (xipos-affected models) from "diverges
at step N>1" (contact_scenario, potentially). This distinction helps
Session 15 (Gate Triage) prioritize fixes: xipos fix unblocks 7 models'
trajectory tests; collision/constraint fix unblocks contact_scenario.

### EGT-4: Free joint quaternion handling in trajectory comparison

`contact_scenario` has a free joint: nq=7 (3 pos + 4 quat), nv=6.
MuJoCo's `mj_integratePos()` normalizes the quaternion after integration.
qpos comparison for this model must handle:
- Quaternion sign ambiguity: q ≡ -q (same orientation)
- qpos indices 3..7 are quaternion components [w, x, y, z]
- qpos indices 0..3 are Cartesian position (direct comparison)

Layer B's `assert_quat_eq()` in `common.rs:211–241` handles this for
single-snapshot FK. Trajectory comparison needs the same handling per step
for the quaternion portion of qpos.

### EGT-5: Ctrl input persistence across steps

MuJoCo keeps `data.ctrl[i]` at the value set by the user until explicitly
overwritten. The gen script sets ctrl once before the stepping loop
(line 157–158):
```python
if ctrl is not None:
    for i, v in enumerate(ctrl):
        data.ctrl[i] = v
```

Then calls `mj_step(model, data)` N times without modifying ctrl. The
Rust test must do the same: set ctrl once, then step N times.

Ctrl values per model:
- actuated_system: `ctrl = [1.0, 0.5]` (motor + position servo)
- composite_model: `ctrl = [1.0]` (motor)
- All others: no ctrl (nu=0 or ctrl stays at zero)

### P9. Chaos-Aware Tolerance Strategy

> Trajectory comparison must account for error accumulation across steps.
> Non-contact systems accumulate FP error linearly; contact systems have
> chaotic sensitivity to small perturbations. The tolerance strategy must
> be justified, not arbitrary.

| Grade | Bar |
|-------|-----|
| **A+** | Two tolerance regimes explicitly defined with justification: (1) **Hinge-only models** (pendulum, double_pendulum, actuated_system, tendon_model, sensor_model, equality_model): `tol(step) = base * (1 + step * growth)` where base and growth are stated with rationale (e.g., base=1e-8 from single-step integration tolerance, growth=0.01 per step from linear FP accumulation in smooth dynamics). (2) **Contact/free-joint** (contact_scenario, composite_model): wider base + faster growth with rationale (e.g., base=1e-6, growth=0.05 because contact activation/deactivation introduces chaotic sensitivity and constraint solver convergence adds noise). Per-field tolerance differentiation documented: qacc may need wider tolerance than qpos/qvel because it includes constraint solver output (iterative solver convergence at 1e-4 vs integration at 1e-8). Tolerance values are NOT tuned to make tests pass — they are derived from algorithmic properties. Note: given EGT-3, most tests will be `#[ignore]`d before tolerances are exercised — but the tolerances must still be correct for when upstream fixes land and tests are un-ignored. |
| **A** | Tolerance regimes defined and justified. Minor gaps in rationale. |
| **B** | Tolerance growth present but arbitrary ("widened until tests pass"). |
| **C** | Flat tolerance or no growth discussion. |

> **Boundary with P1:** P1 grades whether the spec got the MuJoCo step
> pipeline right. P9 grades whether the tolerance strategy for multi-step
> comparison is algorithmically justified and separates chaotic from
> non-chaotic regimes.

### P10. Divergence Diagnostic Quality

> When a trajectory test fails, the diagnostic output must be sufficient
> to identify the root cause without re-running the test with logging.
> This maps trajectory failures back to per-stage root causes.

| Grade | Bar |
|-------|-----|
| **A+** | Diagnostic on failure includes: (1) model name, (2) first step where divergence exceeds tolerance, (3) which field (qpos/qvel/qacc), (4) which DOF index within the field, (5) expected value (MuJoCo reference), (6) actual value (CortenForge), (7) difference magnitude, (8) tolerance at that step. Diagnostic format is specified (exact print format or struct fields). Summary report after all steps: total steps matched before divergence, worst-case field and DOF, accumulated max error. Two divergence patterns distinguished per EGT-3: (a) step-1 divergence — indicates per-stage bug in dynamics pipeline (FK/CRBA/RNE wrong, 7 of 8 models), (b) step-N>1 divergence — indicates collision/constraint sensitivity or accumulation error (contact_scenario, potentially). `#[ignore]` tracking comments include first divergent step and root cause classification (dynamics vs constraint) for Session 15 triage. |
| **A** | Diagnostic includes step, field, DOF, values. Summary present. |
| **B** | Diagnostic prints "mismatch at step N" without field/DOF details. |
| **C** | No diagnostic — test just panics on first mismatch. |

> **Boundary with P2:** P2 grades whether the comparison algorithm is
> complete. P10 grades whether the failure reporting is rich enough to
> diagnose root causes from the output alone.
