# Layer B Per-Stage Reference Comparison — Spec Quality Rubric

Grades the Spec A spec on 11 criteria. Target: A+ on every criterion
before implementation begins. A+ means "an implementer could build this
without asking a single clarifying question — and the result would produce
numerically identical output to MuJoCo."

**MuJoCo conformance is the cardinal goal.** Every criterion in this rubric
ultimately serves conformance. P1 (MuJoCo Reference Fidelity) is the most
important criterion — grade it first and hardest. A spec that scores A+ on
all other criteria but has P1 wrong is worse than useless: it would produce
a clean, well-tested implementation of the wrong behavior.

Grade scale: A+ (exemplary) / A (solid) / B (gaps) / C (insufficient).
Anything below B does not ship.

### EGT-3: Tolerance algorithmic justification

| Stage | Tolerance | Algorithmic property |
|-------|-----------|---------------------|
| FK | `1e-12` | Exact trigonometric/quaternion arithmetic; FP accumulation only through kinematic tree depth (typically 2–6 levels). Pendulum: 1 transform. Double pendulum: 2 transforms. |
| CRBA | `1e-12` | CRBA is exact given FK; same tree traversal depth. Dense M from `mj_fullM()` is a direct unpack. |
| RNE | `1e-10` | Two tree traversals (forward + backward). Coriolis terms involve cross products of spatial velocities, which accumulate FP error. Gravity contributes mass×g×lever — exact for hinge joints at qpos=0. |
| Passive | `1e-10` | Direct computation: spring_force = -stiffness*(qpos-springref), damper_force = -damping*qvel. At qpos0/qvel0, damper=0. Spring forces are exact single multiplications. Looser tolerance covers multi-element models. |
| Collision depth | `1e-6` | Collision detection involves geometric distance computation (sphere-plane, etc.). GJK/MPR algorithms have convergence tolerance. |
| Collision normal | `1e-6` | Normal is the gradient of the distance function — same convergence tolerance as depth. |
| Constraint efc_J | `1e-8` | Constraint Jacobian involves FK-derived transforms plus contact frame rotation. Accumulated FP from FK + contact frame computation. |
| Constraint efc_force | `1e-4` | Iterative solver (Newton/PGS/CG) convergence tolerance. Different solver implementations converge to the same basin but with different residuals. |
| Actuation | `1e-10` | Direct computation: gain*ctrl + bias, then transmission to joint space via gear ratios. |
| Sensor | `1e-8` | Sensors read from upstream stage outputs. Tolerance inherits from the stage that computes the sensed quantity (FK for position sensors, velocity for vel sensors, etc.). |
| Tendon | `1e-10` | Tendon length is a linear combination of joint positions via Jacobian coefficients. Velocity is J*qvel. Both are direct linear algebra. |

### EGT-4: Contact comparison — structural matching required

MuJoCo contact enumeration order depends on the broadphase/narrowphase
processing order, which is geom-pair iteration order in the model. CortenForge
may enumerate contacts in a different order. The reference data includes
`contact_geom_pairs.npy` (int32, shape N×2) for matching.

Strategy: sort both contact sets by (geom1, geom2), match by pair identity,
then compare depth/normal/pos per matched pair. Unmatched contacts (extra or
missing) are test failures.

For `contact_scenario`: 1 contact (sphere-plane), geom pair is deterministic.
For `composite_model`: 2 contacts (two balls on floor), geom pairs are
deterministic but order may differ.

### EGT-5: Constraint Jacobian row ordering

MuJoCo orders constraint rows as: equality constraints first, then friction
loss, then joint limits, then tendon limits, then contacts (normal + friction
rows). Within each group, order follows model definition order.

CortenForge uses the **same ordering** (verified in
`sim/L0/core/src/constraint/assembly.rs:47-53`): equality → friction loss →
joint limits → tendon limits → contacts. This means element-wise comparison
of efc_J/efc_b/efc_force is valid when nefc matches.

For `contact_scenario`: 4 efc rows (1 normal + 3 friction for condim=4 contact
with 1 contact). Shape: (4, 6).
For `equality_model`: 7 efc rows (3 for weld constraint + 1 for joint equality
+ 3 for the weld's rotation part). Shape: (7, 3).
For `composite_model`: 9 efc rows (equality + contact constraints). Shape: (9, 4).

**Primary strategy:** element-wise comparison (since row ordering matches).
**Fallback:** structural matching if nefc differs (CortenForge finds
different number of active constraints).

### EGT-6: CortenForge field name mapping

| MuJoCo C field | CortenForge Rust field | Notes |
|----------------|----------------------|-------|
| `d->xpos` (nbody×3) | `data.xpos: Vec<Vector3<f64>>` | MuJoCo flat array, CortenForge Vec of Vector3 |
| `d->xquat` (nbody×4) | `data.xquat: Vec<UnitQuaternion<f64>>` | MuJoCo [w,x,y,z] matches nalgebra accessor order: `.w`, `.i`, `.j`, `.k`. Internal memory layout differs ([x,y,z,w]) but **comparison uses named accessors, not raw indexing**. Also: q and -q represent the same rotation — **sign ambiguity** must be handled. |
| `d->xipos` (nbody×3) | `data.xipos: Vec<Vector3<f64>>` | Direct match |
| `d->qM` (sparse/dense) | `data.qM: DMatrix<f64>` | MuJoCo sparse, reference uses `mj_fullM()` for dense |
| `d->qfrc_bias` (nv) | `data.qfrc_bias: DVector<f64>` | Direct match |
| `d->qfrc_passive` (nv) | `data.qfrc_passive: DVector<f64>` | Direct match |
| `d->contact[i].pos` (3) | `data.contacts[i].pos: Vector3<f64>` | Direct match |
| `d->contact[i].frame` (9) | `data.contacts[i].normal: Vector3<f64>` | MuJoCo frame[0:3] = normal. CortenForge stores normal separately. |
| `d->contact[i].dist` | `data.contacts[i].depth: f64` | **Sign flip:** MuJoCo `dist` is signed distance (negative = penetration). CortenForge `depth` is positive = penetration. Reference `.npy` stores MuJoCo convention. Comparison: `depth_cf ≈ -ref_dist` |
| `d->contact[i].geom[2]` | `data.contacts[i].geom1/geom2: usize` | MuJoCo int pair, CortenForge usize pair |
| `d->efc_J` (nefc×nv) | `data.efc_J: DMatrix<f64>` | Row ordering matches (per EGT-5/EGT-8) — element-wise comparison valid when nefc matches |
| `d->efc_b` (nefc) | `data.efc_b: DVector<f64>` | Same row ordering as efc_J |
| `d->efc_force` (nefc) | `data.efc_force: DVector<f64>` | Same row ordering as efc_J |
| `d->qfrc_actuator` (nv) | `data.qfrc_actuator: DVector<f64>` | Direct match |
| `d->actuator_force` (nu) | `data.actuator_force: Vec<f64>` | Direct match |
| `d->sensordata` (nsensordata) | `data.sensordata: DVector<f64>` | Direct match — sensor order follows model definition |
| `d->ten_length` (ntendon) | `data.ten_length: Vec<f64>` | Direct match |
| `d->ten_velocity` (ntendon) | `data.ten_velocity: Vec<f64>` | Direct match |

**Critical convention differences:**

1. **Quaternion access:** MuJoCo `.npy` layout is `[w, x, y, z]`. nalgebra
   `UnitQuaternion` accessors: `.w` = scalar, `.i` = x, `.j` = y, `.k` = z.
   Comparison: `ref[i*4+0]` ↔ `q.w`, `ref[i*4+1]` ↔ `q.i`, `ref[i*4+2]` ↔
   `q.j`, `ref[i*4+3]` ↔ `q.k`. No reordering needed — use named accessors.
   **But:** quaternion sign ambiguity (q ≡ -q) means comparison must either
   canonicalize sign (e.g., ensure w > 0) or compare `min(‖q_cf - q_ref‖, ‖q_cf + q_ref‖)`.

2. **Contact depth sign:** MuJoCo `dist` is negative for penetration. CortenForge
   `depth` is positive for penetration. Reference `.npy` stores MuJoCo's signed
   convention. The spec must negate the reference value: `assert_approx(depth_cf, -ref_dist, tol)`.

3. **Contact geom pairs (int32):** Reference `.npy` files for geom pairs use
   `dtype=int32`, but the existing `parse_npy()` only handles float64. The spec
   must either add `parse_npy_i32()` to `common.rs` or use an alternative
   loading strategy.

### EGT-7: Model loading pattern

Tests load models via `sim_mjcf::load_model(xml)` where `xml` is the model
string read from file. No `load_conformance_model()` helper exists. Layer A/D
tests use this pattern directly. The spec should either define a helper
function (preferred for reducing boilerplate) or use the existing pattern.

### EGT-8: Constraint row ordering is deterministic

CortenForge assembles constraint rows in the same order as MuJoCo: equality →
friction loss → joint limits → tendon limits → contacts (verified in
`sim/L0/core/src/constraint/assembly.rs:47-53`, with debug_assert ordering
verification at lines 796-826). Since ordering matches, element-wise comparison
of efc_J/efc_b/efc_force is valid **if** the number of rows (nefc) also matches.
Structural matching is only needed as a fallback if nefc differs.

### EGT-9: Pyramidal vs elliptic cone affects constraint row count

MuJoCo defaults to pyramidal friction cones. CortenForge supports both
(`model.cone == 0` = pyramidal, `model.cone == 1` = elliptic). For pyramidal
contacts, each contact emits `2*(dim-1)` facet rows instead of `dim` rows
(verified in `assembly.rs:162-172`):
- condim=3 (standard friction): 4 rows pyramidal vs 3 rows elliptic
- condim=4: 6 rows pyramidal vs 4 rows elliptic

The reference data was generated with MuJoCo's default (pyramidal). If
CortenForge and MuJoCo use the same cone type, nefc will match. If cone
types differ, nefc will differ and the test will fail at the row-count
pre-check — this is correct behavior (a conformance gap, not a test bug).

The spec must ensure conformance models use the same cone type as MuJoCo's
default (pyramidal). The `contact_scenario.xml` and `composite_model.xml`
models should NOT set `<option cone="elliptic"/>` — they should use the
default pyramidal cone.
