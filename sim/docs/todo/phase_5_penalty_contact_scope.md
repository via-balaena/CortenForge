## 0.5. Architectural finding ŌĆö `NullContact`'s `unimplemented!` is unreachable today

Phase 5 recon caught that `NullContact` ships five `unimplemented!("skeleton phase 2")` stubs (`sim/L0/soft/src/contact/null.rs:19-37`) and yet 171 sim-soft tests run green using `NullContact`. The mechanism: `CpuNewtonSolver` holds the contact model as `_contact: C` (underscore prefix is the suppression) and **never calls a single trait method on it**. The skeleton's *"NullContact carries no state, but the generic is held so Phase-C IPC can slot in"* (`backward_euler.rs:117-118`) was an accurate description of intent at Phase 1 walking-skeleton time, but became architecturally misleading at Phase 4 commit time when the Newton hot path matured into multi-element multi-material assembly without ever needing to grow contact-aware code paths.

**Implication for Phase 5:** the wiring step (commit 5 in ┬¦8) is novel architecture, not a delta on existing wiring. Phase 4's IV-1 passthrough gate verified per-tet material storage; it does *not* verify that contact dispatch produces the same outputs, because contact dispatch did not exist. **V-1 (`contact_passthrough.rs`) is the corresponding gate for Phase 5** ŌĆö the first test to drive `NullContact`'s real-zero-stub methods through the solver and confirm bit-equality with pre-Phase-5 baselines. This is the regression-net spine of the phase.

This finding does **not** indicate prior phase work was wrong. The walking skeleton's architectural commitment (╬│-locked contact-trait-shape) was correct; the comment-versus-reality gap (`_contact: C` underscore prefix not reflected in the docstring claim that contact slots in cleanly) is a doc-tier book finding ineligible for Phase 5 absorption (would mistake a grader-tier code-comment edit for a book-tier amendment). Surface as a gentle followup if the Phase 5 implementation surfaces second-order issues; otherwise leave as-is ŌĆö the comment becomes accurate at commit 5.

**Decision G: `ContactPair` becomes an enum with one variant in Phase 5.**
`pub enum ContactPair { Vertex { vertex_id: VertexId, primitive_id: u32 } }`. Phase H may add `EdgeEdge`/`VertexFace` variants for IPC self-contact; not preempted. **Decision H: Active-pair detection runs once per Newton iter (not per-step).**
Per Newton iter the soft vertices have moved; the active set may have changed. Per-iter recompute is the correct semantic; per-step recompute is an optimization that admits stale-pair bugs and is Phase E performance work, not Phase 5 correctness work. Cost: one O(`n_vertices ├Ś n_primitives`) walk per iter. Phase 3-class meshes (~3000 tets at canonical sphere ŌåÆ ~600ŌĆō1000 vertices on the BCC lattice) ├Ś 1 primitive (`RigidPlane`) gives ~1000 SDF evaluations per iter; Phase 5 V-3 finest-level mesh (Open Decision ┬¦9 ŌĆö release-mode-only; ~25k tets at smallest cell size ŌåÆ ~5ŌĆō10k vertices) ├Ś 1 primitive gives ~10k SDF evals per iter; both negligible.

**Decision I: Vertex-vs-`RigidPlane` only in Phase 5 (no softŌåösoft self-contact, no planeŌåöplane, no vertex-vs-`RigidSphere`).**
