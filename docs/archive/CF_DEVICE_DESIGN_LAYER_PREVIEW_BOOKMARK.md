# cf-device-design Layer-Preview Quality — Bookmark

**Status**: OPEN. Layer-surface previews are visibly chunky on the iter-1
sock-over-capsule fixture; cast-quality is unaffected (cf-cast-cli samples
the cleaned STL's SDF directly, not this proxy). Bookmarked 2026-05-16 as
a coupled multi-system arc rather than a single-commit polish task.

## What the user sees

In cf-device-design with the iter-1 cleaned scan loaded:

- **Cavity (3 mm inward inset)**: slightly more chunky / faceted than the
  scan but visibly clean — readable as "low-poly proxy preview."
- **Layer 0 (10 mm outward thickness from cavity)**: noticeably chunky.
  Facet edges visible.
- **Layer 1 (3 mm beyond Layer 0, = 10 mm outside scan)**: bad. Wrinkled
  / faceted enough that the layer's shape doesn't match the smooth offset
  the user expects.

User comment 2026-05-16: layer 0 was "a very real concern," layer 1 was
"bad."

## Why it happens

Layer + cavity surfaces are built from a shared `EnvelopeProxyMesh`
(~1500 faces, decimated from the cleaned STL's ~169k via
`compute_envelope_proxy_mesh` → topology-preserving simplify). Per-vertex
displacement is `proxy.vertex_radial_directions[i] * offset_m`, where the
radial direction is computed as `(vertex - nearest_centerline_pt).normalize()`
(split by Z-region; below `centerline_min_z` the Z component is zeroed
to pin the bottom rim).

Two coupled failure modes:

1. **Coarse per-vertex radials (1500-face proxy)**: with only 1500
   vertices spanning a 130 mm-tall body, adjacent vertices average over
   sparse incident-face neighborhoods; their radials can diverge by
   several degrees. A 10 mm outward displacement amplifies that
   divergence into visible mm-scale triangle skew — the chunky facets
   on Layer 0/1.

2. **Wiggle in the centerline polyline's tip end (dome)**: cf-scan-prep's
   `cross_section_centroids` centerline algorithm goes unstable as
   cross-sections shrink toward a point at the dome tip. The last few
   centerline points wander laterally from the body's true axis. **All**
   dome vertices use the centerline's TOP endpoint as their
   `nearest_centerline_pt` — so a wandering endpoint means dome radials
   point away from a wrong center. Wiggle ≈ 2-3 mm at the tip.

The two failure modes COUPLE: a coarse proxy hides the wiggle (few dome
verts → wiggle-induced radial differences rarely land on adjacent
triangles). A denser proxy exposes it (more dome verts → many adjacent
pairs straddle the wiggle's lateral offset → adjacent radials diverge →
triangles fold under displacement → back-faces render dark through
`cull_mode: None`). Verified empirically 2026-05-16 by raising the proxy
to 8000 faces: the chunkiness flattened in the body region but the dome
gained widespread dark "dappling" patches that weren't there at 1500.

## Why this isn't fixed now

The single-knob-to-tune answer (raise proxy face count) is blocked by the
centerline wiggle. Fixing one without the other regresses the other.
This is a multi-system arc requiring coordinated changes in two crates;
worth its own dedicated session rather than rushed mid-iter-1.

## Cast impact

**None**. cf-cast-cli does not consume `EnvelopeProxyMesh` or any layer
surface generated by cf-device-design. It reads `cavity.inset_mm` +
`layers[*].thickness_mm` from `.design.toml` and samples the cleaned
STL's SDF directly at the configured offsets. The cast geometry is
accurate at the 2 mm SDF cell resolution regardless of how the
preview-only layers look.

Workshop iter-1 cast can proceed against the current preview without
concern.

## Suggested attack order (next session)

**Phase 1 — fix the centerline wiggle in cf-scan-prep** (independent
prerequisite):

- (a) Trim the last K cross-section centroids that fall below an
  inscribed-circle radius threshold (the failure mode is small-circle
  centroid noise). Cheap, single-commit.
- (b) Smooth the centerline polyline tail via a moving-average or
  Laplacian-on-polyline pass. Larger change but more general.
- (c) Switch to a Frenet-frame walker for the last 10-20 mm. Most
  rigorous, biggest change.

Recommend (a) for the first attempt; ship + visually verify; escalate
only if needed.

**Phase 2 — retry raising cf-device-design's proxy density**:

- Once the centerline endpoint sits on the body's true axis, retry
  `ENVELOPE_PROXY_TARGET_FACES = 8000`. Expected outcome: clean dome
  (no dappling) + smooth body.

**Phase 3 — independent layer-mesh smoothing** (optional, second
backstop):

- Add a 2-3 iteration Laplacian smoothing pass on the displaced layer
  mesh, AFTER `build_displaced_proxy_mesh_with_colors`. Cheap; would
  also mask any residual coupling artifacts from Phase 1's imperfections.

**Phase 4 — replace the proxy approach** (deferred; only if Phases 1-3
prove insufficient):

- Sample the cleaned STL's SDF at each layer's offset distance,
  marching-cubes a fresh mesh per-layer on slider tick. Geometrically
  correct (no per-vertex radial direction at all); cost is per-tick
  marching-cubes. Profile first.

## Trigger condition for un-bookmarking

The iter-1 physical cast surfaces a real problem traceable to layer
preview quality (e.g., the user designs against a misleading visual and
the cast comes out wrong). Or: post-iter-1, ahead of workshop iter-2,
when there's a dedicated session for design-tool polish.
