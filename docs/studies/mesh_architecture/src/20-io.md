# mesh-io — STL, OBJ, PLY, 3MF, STEP and the custom-attribute path

What this part covers (when authored to depth):

## Scope and tier

`mesh-io` is L0-io tier — depends on `mesh-types` and one parser library per format, exposes `load_*` / `save_*` per format plus generic `load_mesh` / `save_mesh` with extension-based format detection. Zero Bevy dependencies.

Five formats supported as of the skeleton:

| Format | Library | Variants | Custom attrs | Default? |
|---|---|---|---|---|
| **STL** | hand-rolled | binary, ASCII | None (positions+normals only) | always |
| **OBJ** | hand-rolled | ASCII | None (positions+UVs) | always |
| **PLY** | `ply-rs` | binary little-endian, ASCII, BE | **Yes** (per-vertex / per-face properties) | always |
| **3MF** | `zip` + `quick-xml` | XML-in-zip | Multi-part assemblies, materials | feature-gated (`threemf`) |
| **STEP** | `truck-stepio` family | STEP/STP | Parametric solid model | feature-gated (`step`) |

The two feature-gated formats keep heavy dependencies out of the default graph (zip+zstd+cc for 3MF; the truck CAD kernel for STEP).

## Format tradeoffs

The depth pass covers per-format tradeoffs in detail. The skeleton-level summary:

- **STL** — universal in 3D printing, simplest format, no attributes beyond face normals. Use for: fab handoff, tooling integration, lowest-common-denominator interchange. Don't use for: per-vertex attribute output, multi-part assemblies, parametric geometry.
- **OBJ** — text-based, supports vertex/UV/normal data, optional materials via `.mtl` sidecar. Use for: legacy game-engine pipelines, simple text-readable mesh snapshots. Don't use for: large meshes (text encoding inflates size), per-vertex custom attributes.
- **PLY** — Stanford Triangle Format, binary or ASCII, **first-class custom property support**. Use for: per-vertex attribute snapshots (the soft-body simulator's primary output), MeshLab/ParaView interop, scientific visualization. The natural format for "attribute-rich mesh frame."
- **3MF** — modern multi-part assembly format, XML in a zip. Use for: multi-piece molds, multi-material assemblies, integrated metadata (manufacturer, units). Don't use for: per-vertex scalar field snapshots (3MF's strength is part-level, not vertex-level).
- **STEP** — parametric CAD interchange, full B-rep solid model. Use for: handing parts to traditional CAM workflows, mold fab via CNC. Don't use for: triangle-mesh interchange (STEP isn't natively triangulated).

## Format detection and the umbrella functions

Path-based extension detection via `MeshFormat::from_path` plus `load_mesh` / `save_mesh` umbrella functions. Format-specific functions (`load_stl`, `save_ply`, etc.) remain available for callers who want to be explicit. The umbrella functions are convenience for the common case of "I have a path, infer the format."

The depth pass covers the format-detection edge cases (case-insensitive extension matching, the `.stp` vs `.step` synonym, future format additions like VTK or USD).

## The PLY custom-attribute path

PLY is the format where custom per-vertex attributes are first-class. The PLY header declares each vertex property with a name and type:

```
element vertex 1024
property float x
property float y
property float z
property float displacement_magnitude   ← custom
property float von_mises_stress          ← custom
property uchar material_id               ← custom (cast to u8)
```

ParaView, MeshLab, Blender (with addons), and pyvista all read these custom properties and let the user select one for colormap visualization. This makes PLY the canonical format for "snapshot a deformed mesh with attached physics fields."

The geometry-only path is `save_ply(&IndexedMesh, path, binary)` / `load_ply(path) -> IndexedMesh`. It writes and reads positions plus face indices; everything else on disk is dropped. That path remains available for callers that want the simplest contract.

The attributed path is `save_ply_attributed(&AttributedMesh, path, binary)` / `load_ply_attributed(path) -> AttributedMesh`. It round-trips every closed slot on `AttributedMesh` and the open-ended `extras` map, using PLY-canonical property names so external tools recognize the data without configuration.

### Property-name table

| `AttributedMesh` slot | PLY properties | PLY type | Written when |
|---|---|---|---|
| `geometry.vertices` | `x, y, z` | `float` | always |
| `normals` | `nx, ny, nz` | `float` | `Some(_)` |
| `colors` | `red, green, blue` | `uchar` | `Some(_)` |
| `zone_ids` | `zone_id` | `uint` | `Some(_)` |
| `clearances` | `clearance` | `float` | `Some(_)` |
| `offsets` | `offset` | `float` | `Some(_)` |
| `uvs` | `s, t` | `float` | `Some(_)` |
| `extras["k"]` | `k` | `float` | iterated in `BTreeMap` order |

Property declaration order in the header — and equivalently the per-vertex byte order in the binary body — is fixed: positions, then closed slots in the table order above, then `extras` keys in `BTreeMap` iteration order. Iteration determinism is load-bearing for byte-stable goldens (matches the commit-1 `BTreeMap`-over-`HashMap` precedent on `AttributedMesh::extras`).

### Reserved-name validation

Names that would collide with the canonical slots above — including `alpha` (PLY-reserved alongside red/green/blue) — cannot be used as `extras` keys for a PLY write. Validation also rejects empty names, names containing whitespace, names containing non-printable bytes, and PLY-format keywords (`ply`, `format`, `comment`, `element`, `property`, `end_header`, `obj_info`).

Validation runs at write time in `mesh-io`, **not** at `insert_extra` time in `mesh-types`. This is intentional: `mesh-types` stays insulated from per-format naming rules, so a future `save_obj_attributed` or `save_3mf_attributed` can apply its own ruleset without forcing `AttributedMesh` to anticipate every format's reserved set. The cost is that "is this `extras` map writable to PLY?" is answered at the I/O boundary; the benefit is that a canonical name like `nx` is never silently shadowed.

`save_ply_attributed` returns `IoError::InvalidAttributeName { name, reason }` for any of these failures and creates no file on disk. Slot-length mismatches (e.g., `mesh.normals.as_ref().unwrap().len() != mesh.vertex_count()`) similarly return `IoError::InvalidContent` before any bytes are written. `AttributedMesh` slots are public fields, so the boundary check guards against direct-mutation invariant violations that bypass `insert_extra`'s length validation.

### Loader semantics

`load_ply_attributed` recognizes the canonical names above and pulls them into the matching closed slots. Two rules govern everything else:

1. A canonical set is populated only when **all** its component property names are present (`nx, ny, nz` together, `red, green, blue` together, `s, t` together). A partial set is dropped — and because the partial names are reserved, they do not fall through into `extras` either.
2. Any property whose name is **not** reserved and whose type is `float` or `double` is collected into `extras` under its raw name (`double` is narrowed to `f32`). Properties of other types (`int`, `uchar` outside the RGB triple, etc.) are silently ignored. This keeps the loader forward-compatible with externally-authored PLY files that mix in non-float fields.

The geometry-only `load_ply` is unchanged; it still drops everything except positions and faces.

### What's still NOT covered

- **Per-face attributes.** `AttributedMesh` is per-vertex by design; PLY supports per-face properties but no `AttributedMesh` slot consumes them. Future extension if a use case arises.
- **Floating-point colors.** Some PLY files store `red, green, blue` as `float` in [0, 1]. The current loader only accepts `uchar` (matching `VertexColor`'s `u8` fields); float-typed RGB properties are dropped. A future `from_float`-style coercion path is straightforward but unjustified by current consumers.
- **Integer extras.** `extras` is `Vec<f32>`; integer-typed custom properties on external PLY files are silently dropped on load. The writer can never produce integer extras.
- **Streaming I/O.** Same scope note as the rest of `mesh-io`: loaders parse the entire file into memory before returning.

## What's NOT in `mesh-io`

- **Custom format definitions.** If a user wants a CortenForge-specific binary format, they roll it themselves (or ask for it as a feature add).
- **Streaming I/O.** All current loaders parse the entire file into memory before returning. For multi-gigabyte meshes this would matter; current platform target is benchtop scale (~30k DOFs / ~50k triangles), well within memory.
- **Compression for STL / OBJ.** Both are uncompressed at the format level. 3MF gets zip compression for free.
- **Per-face attributes.** Some formats support them (PLY, OBJ via face groups); the current API is per-vertex-attribute focused. Per-face attributes are a future extension if a use case arises.
- **Texture resource loading.** OBJ's `.mtl` sidecar references texture files; current OBJ loader skips these (positions+UVs+faces only).

The depth pass enumerates these explicitly so future readers know what's intentional vs accidental.
