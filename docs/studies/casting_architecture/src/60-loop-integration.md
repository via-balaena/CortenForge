# Integration with the design loop

What this part covers (when authored to depth):

This is the load-bearing part — where casting stops being an after-the-fact check and starts being a constraint that participates in design optimization.

## Manufacturability as a reward term

The soft-body book's [`RewardBreakdown`](../../soft_body_architecture/src/100-optimization/00-forward.md) carries four scalar terms today: pressure_uniformity, coverage, peak_bound, stiffness_bound. Casting adds a fifth (and possibly more):

```rust
pub struct RewardBreakdown {
    pub pressure_uniformity: f64,
    pub coverage: f64,
    pub peak_bound: f64,
    pub stiffness_bound: f64,
    // New from sim-cast:
    pub manufacturability: f64,  // aggregate score from ManufacturabilityReport
}
```

The aggregate score is smooth in design parameters when the underlying constraint surfaces are smooth (wall thickness, draft, material volume). It's discontinuous when topology changes (a parting surface that worked starts trapping the cast under perturbation). Discontinuities map onto barrier functions or onto explicit topology-change classifiers — same machinery the simulator uses for [`EditResult`](../../soft_body_architecture/src/100-optimization/00-forward.md).

## Manufacturability gradients

The optimizer wants `∂R_manufacturability / ∂θ`. Different constraint terms give it differently:

- **Wall thickness, draft, surface area.** Smooth functions of geometry. Standard FD or analytic gradients.
- **Demoldability.** Discrete pass/fail under topology changes. FD wrapper or differentiable relaxation.
- **Material cost, cure time.** Linear in casted volume. Closed form.
- **Sequence validity (does the multi-shot ordering exist?).** Combinatorial. No useful gradient; classifier flag instead.

The composition is straightforward when each term contributes additively to the aggregate score. The interesting case is when a discontinuous term *gates* a continuous term — e.g., manufacturability is zero everywhere a valid casting sequence exists, NaN where it doesn't. Optimizer handles NaN gates via the `EditResult` classifier per the soft-body book's existing pattern.

## Sim-to-real for cast parts

Cast parts deviate from the simulator's model in predictable ways:

- **Shrinkage.** 0.1–0.5% for silicones, 1–2% for polyurethanes, isotropic on first order, anisotropic when cure conditions are non-uniform.
- **Surface roughness.** The mold's interior surface texture transfers to the cast. Mold-side defects (printer layer lines, machining marks) become cast-side features.
- **Bubbles.** Stochastic. Visible as voids in the cured cast; affect mechanical properties (less material = lower stiffness in voided regions).
- **Cure variation.** Inhomogeneous cure (one shot of the mold is cooler than another) produces stiffness variation the simulator doesn't model.

Each of these is a sim-to-real residual term. Same machinery as [soft-body Part 10 Ch 05 sim-to-real](../../soft_body_architecture/src/100-optimization/05-sim-to-real.md) — a residual GP trained on physical measurements, applied as a corrected reward at design-eval time. Cast-specific because the residual sources (shrinkage, bubbles, cure variation) are casting-specific phenomena.

## The full loop, with casting

The soft-body book's [Part 10 Ch 06 full loop diagram](../../soft_body_architecture/src/100-optimization/06-full-loop.md) extends naturally:

```text
INNER LOOP (sim-side):
    cf-design → SDF + MaterialField
    sim-soft → R_sim_physics
    sim-cast → R_sim_manufacturability
    combined R_sim = R_sim_physics + R_sim_manufacturability
    sim-to-real correction → R̂_real(θ)
    BayesOpt → next θ

OUTER LOOP (physical, weeks):
    shortlist K candidates
    ⟶ cast K parts (the casting plan generated by sim-cast)
    measure (force, pressure, AND cast-quality: bubbles, dimensional drift)
    rate
    update sim-to-real residual GP (now over physics AND manufacturability terms)
    update preference GP
    restart inner loop
```

The casting integration adds two things to the loop:

1. A reward term that the optimizer reads each inner-loop step.
2. A casting plan that gets emitted at shortlist time, telling the human (or robot) how to fab the K candidates.

## Programmatic surface

`sim-cast::ForwardManufacturability` mirrors `sim-soft::ForwardMap` — same shape, same `evaluate`/`gradient` split, same `Tape` plumbing for autograd. The optimizer composes the two via additive reward.

## Open questions

- The right weight between physics reward and manufacturability reward. Probably preference-learned (the human rates batches under both lenses), not hand-tuned.
- Should sim-cast write to the same tape sim-soft writes to (one combined gradient pass), or compute its own gradient and add at the reward level (separate passes, summed)? Performance vs. composability tradeoff.
