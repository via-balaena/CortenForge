# Conjugate Gradient

Preconditioned Conjugate Gradient is the canonical iterative solver for symmetric positive-definite systems — the algorithm Hestenes and Stiefel published in 1952, now textbook material in [Saad 2003](../../appendices/00-references/07-solvers.md#saad-2003). For SPD Hessians it converges in $O(\sqrt{\kappa})$ iterations where $\kappa$ is the condition number; every iteration is one SpMV plus O(DOFs) vector arithmetic. The [Ch 02 parent's Claim 1](../02-sparse-solvers.md) commits it as the primary GPU solver for `sim-soft`; this leaf specifies the algorithm, the warm-start pattern, the termination criterion, and the `SpMvOp` trait boundary that makes BSR and matrix-free interchangeable under the solver.

## The algorithm

Preconditioned CG, standard Hestenes-Stiefel form, operating on $H x = b$ with SPD preconditioner $M \approx H^{-1}$:

```text
Initialize: r_0 = b - H x_0 ;  z_0 = M r_0 ;  p_0 = z_0
for k = 0, 1, 2, ...
  α_k   = ⟨r_k, z_k⟩ / ⟨p_k, H p_k⟩         # scalar
  x_k+1 = x_k + α_k p_k                        # AXPY
  r_k+1 = r_k - α_k H p_k                      # AXPY
  if ‖r_k+1‖ ≤ tol : return x_k+1
  z_k+1 = M r_k+1                              # preconditioner apply
  β_k   = ⟨r_k+1, z_k+1⟩ / ⟨r_k, z_k⟩        # scalar
  p_k+1 = z_k+1 + β_k p_k                      # AXPY
```

Per iteration on GPU ([Ch 00 §00 kernel-types](../00-wgpu-layout/00-kernel-types.md) taxonomy): one SpMV kernel (per-DOF shape), one preconditioner apply (variable shape per [§03](03-preconditioning.md)), three AXPY kernels (per-DOF; $x \mathrel{+}= \alpha p$, $r \mathrel{-}= \alpha H p$, $p \leftarrow z + \beta p$), two dot-product reductions (per-DOF tree reduction, 2 dispatches per reduction). Total ≈8–10 dispatches per CG iteration depending on preconditioner apply count; [Ch 00 §02 async](../00-wgpu-layout/02-async.md)'s "≈7 dispatches/iter" figure is a simplified ballpark that undercounts the preconditioner apply and the p-update AXPY — the Phase-E aggregate kernel-per-Newton count still lands in the right order of magnitude but the per-iter breakdown here is the precise one.

## The SpMvOp trait abstraction

The inner loop's SpMV is behind a trait, not a concrete buffer reference:

```rust
pub trait SpMvOp {
    fn apply(&self, input: &GpuTensor<f32>, output: &mut GpuTensor<f32>);
    fn dim(&self) -> usize;
}

pub struct CgSolver<'a> {
    operator:      &'a dyn SpMvOp,                  // H — BSR or matrix-free
    precond:       &'a dyn Preconditioner,          // M — see §03
    tolerance:     f64,
    max_iter:      usize,
    workspace:     GpuCgWorkspace,                  // r, z, p, H·p buffers
}
```

The solver itself is format-agnostic. [Ch 01 §01 BSR](../01-sparse-matrix/01-bsr.md) provides a `BsrSpMvOp`; [Ch 01 §02 custom](../01-sparse-matrix/02-custom.md) provides a `MatrixFreeSpMvOp`. Swapping between them is a runtime decision from the chassis's memory-budget heuristic; the CG code doesn't see the swap.

This is the same trait-boundary pattern `sim-ml-chassis` uses everywhere — Algorithm, VecEnv, Policy — and the reason CG "survives" the GPU port as a structural copy of the CPU implementation: the CPU `cg_cpu.rs` and GPU `cg_gpu.rs` differ only in their `SpMvOp` implementation, not in the iteration logic.

## Warm-start from the previous Newton iterate

For the forward Newton loop, each CG call solves $H^{(k)}\, \Delta x^{(k)} = -g^{(k)}$ for the next Newton step. Successive Newton iterations have similar $H$ (elastic and contact Hessians change smoothly between iterates as long as the active-contact-set is stable) and similar RHS (gradient shrinks as Newton approaches the minimum). The previous iterate's $\Delta x$ is a decent initial guess for the next:

```rust
pub fn newton_step(solver: &mut CgSolver, H: &dyn SpMvOp, g: &GpuTensor<f32>,
                   prev_dx: &GpuTensor<f32>) -> Result<GpuTensor<f32>, CgError> {
    let mut dx = prev_dx.clone();                              // warm start
    solver.solve(H, &g.negate(), &mut dx)?;                    // CG in-place
    Ok(dx)
}
```

Warm-starting typically cuts CG iteration count by 2–5× on the canonical scene — the Newton-close region has $H^{(k+1)} \approx H^{(k)}$ and $\Delta x^{(k+1)} \approx \Delta x^{(k)}$, so CG has to refine rather than discover. The [Ch 02 parent's Claim 2](../02-sparse-solvers.md) generalizes this warm-start pattern to the IFT backward pass: each backward solve gets the previous backward iterate as its initial guess, recovering the "one-factor-many-RHSes" cost shape in iterative-solver form.

## Termination and the mixed-precision guard

CG terminates when $\|r_k\|_2 \leq \text{tol}$. The tolerance matches the outer Newton tolerance — no point running CG to machine precision inside a Newton step whose residual check is already looser:

```rust
pub fn adaptive_cg_tol(newton_residual: f64) -> f64 {
    (newton_residual * 0.1).max(1e-10)         // inner loop 10× tighter than outer
}
```

Mixed precision per [Ch 02 parent Claim 3](../02-sparse-solvers.md): SpMV and preconditioner in `f32`; dot-product accumulators, scalar coefficients $\alpha, \beta$, and the solution vector in `f64`. WGSL does not guarantee `f64` natively — most backends require extensions — so the `f64` accumulator on the default wgpu backend is emulated: the partial-sum reduction runs in `f64` inside each workgroup (using a compensated-summation pattern), and the final cross-workgroup reduction lands on CPU for a handful of scalars. Where a backend does provide `f64` storage (Vulkan `VK_KHR_shader_float64`, Metal 2.4+), the native path is used; the abstraction hides the choice. Either way, the cost is a small fraction of the SpMV wall-clock — reductions are not the inner-loop rate-limiter.

One iterative-refinement step at `f64` SpMV follows convergence per [Carson & Higham 2018](../../appendices/00-references/07-solvers.md#carson-higham-2018): recompute $r = b - H x$ at full precision, solve $H e = r$ via the same `f32`-preconditioned CG for the correction $e$, update $x \leftarrow x + e$. The refinement step re-rounds the final iterate to match the 5-digit [gradcheck threshold](../../110-crate/04-testing/03-gradcheck.md) Phase E's regression suite expects. Cost ≈10% extra wall-clock per Newton step.

## CG stagnation detection

CG can stagnate — iterate count hits the cap without reaching tolerance. Two failure modes matter:

**Loss of orthogonality.** After ≈$\sqrt{\kappa}$ iterations the conjugate directions drift from true H-orthogonality due to floating-point accumulation. On well-preconditioned problems this happens well after convergence; on poorly-preconditioned problems (Jacobi at 300k DOFs) it can happen before convergence. `sim-soft`'s CG flags this by detecting $\langle p_k, H p_k \rangle \leq 0$ (a negative-curvature signal) and triggers a MINRES retry ([§01](01-minres.md)).

**Indefinite $H$.** Outside the IPC barrier's basin ([Part 5 Ch 00 Claim 2](../../50-time-integration/00-backward-euler.md)), the Hessian loses positive-definiteness and the CG recursion diverges. Same detection (negative curvature on the Krylov direction), same fallback (MINRES on the same matrix).

Both failures are recorded on the [Ch 03 autograd tape](../03-gpu-autograd.md) as a solver-status entry per Newton iteration, so downstream consumers — [Part 10's optimizer](../../100-optimization/00-forward.md) weighting stagnated gradients, the [adaptive-$\Delta t$ halving decision](../../50-time-integration/02-adaptive-dt.md) from Part 5 Ch 02 — can inspect the flag and adjust.

## What this sub-leaf commits the book to

- **Preconditioned CG (Hestenes-Stiefel form) is the Phase E primary solver.** SPD matrices only; indefinite fallback is MINRES per [§01](01-minres.md). Standard textbook algorithm from [Saad 2003 Ch 6](../../appendices/00-references/07-solvers.md#saad-2003).
- **The `SpMvOp` trait is the format boundary.** BSR and matrix-free are interchangeable under the solver; the CG iteration logic is format-agnostic.
- **Warm-start from the previous Newton iterate cuts iteration count by 2–5×.** The "similar $H$, similar RHS" between successive Newton steps means the previous $\Delta x$ is a decent initial guess.
- **Mixed precision: `f32` SpMV + preconditioner, `f64` accumulators + solution.** `f64` reductions use compensated summation inside each workgroup plus a small cross-workgroup reduction (on CPU by default; natively on backends exposing `f64` storage extensions). Iterative refinement at `f64` SpMV per [Carson & Higham 2018](../../appendices/00-references/07-solvers.md#carson-higham-2018) recovers the 5-digit gradcheck precision.
- **Stagnation and negative-curvature detection trigger MINRES fallback.** Detection is a cheap test on $\langle p_k, H p_k \rangle$; the failure flag is recorded on the autograd tape.
