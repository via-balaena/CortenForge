# Gradcheck passing

Phase D's second first-working guarantee is that the gradcheck suite from [Part 11 Ch 04 §03](../../110-crate/04-testing/03-gradcheck.md) goes green on every PR from this phase onward. Concretely: the γ-locked [`ForwardMap::evaluate`'s determinism-in-θ contract](../../100-optimization/00-forward.md) holds bit-equally across repeated calls, the [Part 6 Ch 02 IFT adjoint gradient](../../60-differentiability/02-implicit-function.md) agrees with central-difference finite differences to 5 digits on a 100-tet neo-Hookean cube at static equilibrium, and [`GradientEstimate::Exact`](../../100-optimization/00-forward.md) is the only variant Phase D scenes return. This sub-leaf names what that green-ness means at the capability level; [Part 11 Ch 04 §03](../../110-crate/04-testing/03-gradcheck.md) names the tests themselves.

## What activates at Phase D

Three gradcheck rows go live at Phase D, across the assertions operationalized in [Part 11 Ch 04 §03](../../110-crate/04-testing/03-gradcheck.md):

- **Determinism-in-θ (§01).** [Part 11 Ch 04 §03's determinism-in-θ test](../../110-crate/04-testing/03-gradcheck.md) asserts `forward_map_is_deterministic_in_theta` — two `evaluate(theta, tape)` calls at the same θ produce bit-equal `RewardBreakdown`s, modulo cache state. The full `ForwardMap` contract lands at Phase D (Phases A–C exercise lower-layer determinism via [Part 11 Ch 04 §00 unit tests](../../110-crate/04-testing/00-unit.md) but cannot bind the end-to-end `ForwardMap` surface until `readout/` and the [`sim-ml-chassis` coupling](../../110-crate/02-coupling/03-ml-chassis.md) close), so §01 activates here and not earlier.
- **FD-vs-IFT 5-digit agreement on composed scalar reward (§02).** [Part 11 Ch 04 §03's analytic-gradient agreement test](../../110-crate/04-testing/03-gradcheck.md) runs on the 100-tet neo-Hookean cube fixture. The forward solve records the faer `Llt<f64>` factor-on-tape per [Part 5 Ch 00](../../50-time-integration/00-backward-euler.md); the IFT adjoint from [Part 6 Ch 02](../../60-differentiability/02-implicit-function.md) retrieves it and applies one back-substitution against $-\partial r/\partial\theta$. The test asserts `rel_tol = 1e-5` agreement against central-difference FD on the composed scalar reward. A proto-form runs from [Phase B](../../110-crate/03-build-order.md#the-committed-order) on the elastic-only solver; Phase D is where the full `ForwardMap` surface including contact and readout passes.
- **`GradientEstimate::Exact` asserted on every Phase D scene.** The [95/5 split from Part 10 Ch 00](../../100-optimization/00-forward.md) binds operationally here. Phase D's synthetic 2-parameter design space stays inside a fixed mesh topology — no `TopologyChanging` edits, no [adaptive-refinement events](../../70-sdf-pipeline/03-adaptive-refine.md), no [adaptive-Δt shrinks](../../50-time-integration/02-adaptive-dt.md) on the canonical problem's timestep — so every returned `GradientEstimate` must match `Exact`. A PR that silently starts returning `Noisy { variance }` on a smooth scene fails the companion assertion in [Part 11 Ch 04 §03](../../110-crate/04-testing/03-gradcheck.md). The `Noisy` variance-calibration path activates at [Phase G](../04-sdf.md) when live re-meshing fires its first topology crossing.

## Why this is a capability milestone, not an implementation detail

Three phases ship gradcheck rows before Phase D per [Part 11 Ch 04 §03's Phase landing schedule](../../110-crate/04-testing/03-gradcheck.md): Phase A tests [`material/`](../../110-crate/00-module-layout/00-material.md) and [`mesh/`](../../110-crate/00-module-layout/02-mesh.md) in isolation before consumer modules compile, Phase B tests the elastic-only solver with a 100-tet FD-vs-IFT check, Phase C tests contact on a reduced-dimensional scene. Each earlier row is a lower-layer correctness check; none binds the full `ForwardMap` contract the optimizer consumes.

Phase D is the first phase where the gradcheck suite binds the γ-locked [`ForwardMap` contract](../../100-optimization/00-forward.md) end-to-end — determinism-in-θ plus smooth-scene `Exact` gradient agreement plus 5-digit FD-vs-IFT on the composed scalar reward. A PR that breaks the forward map's contract from this phase forward fails CI rather than being caught later by an optimizer-convergence regression. This is what makes Phase D the "first working" line: not that the math is right for the first time — it was right at Phase B — but that it is *contractually tested at the surface downstream consumers depend on*.

The downstream consequence is load-bearing. [Part 10 Ch 02's BayesOpt GP](../../100-optimization/02-bayesopt.md) caches training data keyed on θ, and [Part 10 Ch 05's residual-GP online updates](../../100-optimization/05-sim-to-real.md) compose against the same cached evaluations. Both depend on the cached data at θ remaining valid across calls at the same θ. The §01 determinism test catches a drift that would silently invalidate either path before the drift reaches them.

## What gradcheck does not yet test at Phase D

- **CPU-vs-GPU 5-digit regression.** [Phase E](../03-gpu.md), per [Part 11 Ch 04 §03's CPU-vs-GPU row](../../110-crate/04-testing/03-gradcheck.md). The [`Solver` trait's associated `Tape` type](../../110-crate/01-traits/00-core.md) is designed to carry either a CPU or GPU representation; at Phase D only the CPU half exists, so the paired-solver regression is structurally in place but dormant. The bridge is intentional — Phase E's GPU port fills in the second tape without changing the harness.
- **`Noisy { variance }` variance-calibration.** [Phase G](../04-sdf.md), when [`TopologyChanging` edits](../../70-sdf-pipeline/04-live-remesh.md) fire `Noisy { variance }` for the first time on a real design sweep. Phase D's synthetic design space stays inside a fixed mesh topology; the [FD-wrapper through a topology crossing from Part 6 Ch 05](../../60-differentiability/05-diff-meshing.md) has no input to receive until `sdf_bridge/` ships. The "reported variance is a calibrated upper bound on true FD noise" assertion in [Part 11 Ch 04 §03](../../110-crate/04-testing/03-gradcheck.md) is dormant until then.
- **Physical-print residual validation.** Post-Phase-I per [Ch 07's open-questions housing](../07-open-questions.md). The [`SimToRealCorrection::correct`](../../100-optimization/05-sim-to-real.md) method and the [`MeasurementReport` ingestion path](../../100-optimization/05-sim-to-real.md) are part of the `sim-opt` boundary at Phase D, but the residual GPs carry no measurement data yet. Validating corrected-against-measured on held-out θ is the post-Phase-I loop's own test class with a measured-curve oracle, not an analytic-gradient oracle; gradcheck does not absorb it.

## What Phase D locks in

Phase D installs the gradcheck harness at the `ForwardMap` surface and commits to running it on every PR from this phase onward. [Phase E](../03-gpu.md), [Phase G](../04-sdf.md), [Phase H fidelity upgrades](../../110-crate/03-build-order.md#the-committed-order), and [Phase I visual layer](../05-visual.md) extend the suite without weakening the Phase D baseline. A PR that touches [`material/`](../../110-crate/00-module-layout/00-material.md), [`contact/`](../../110-crate/00-module-layout/03-contact.md), [`solver/`](../../110-crate/00-module-layout/04-solver.md), or [`autograd/`](../../110-crate/00-module-layout/07-autograd.md) on any subsequent phase runs the full suite — §01 + §02 at a minimum, §03 added at Phase E — on the small reference scene before it merges. Gradcheck is not a Phase D gate that lifts once passed; it is the always-on correctness backbone [Part 11 Ch 04 §03's continuous-gradcheck commitment](../../110-crate/04-testing/03-gradcheck.md) names.
