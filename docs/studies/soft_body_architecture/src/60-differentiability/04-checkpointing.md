# Checkpointing strategies

The [time adjoint](03-time-adjoint.md) integrates backward through the forward trajectory; to do that it needs access to the forward state $x_t$ at every step where the adjoint ODE evaluates the Jacobian $\partial f / \partial x$. The naive implementation stores every $x_t$ and every factorization along the forward trajectory — linear memory in the number of timesteps $T$. At the design-mode resolution (~30k DOFs with factorizations at hundreds of megabytes each) a multi-hundred-to-thousand-step rollout allocates into the tens-to-hundreds of gigabytes before the backward pass has started. This chapter names the three options for surviving that memory bill.

| Section | What it covers |
|---|---|
| [Uniform checkpointing](04-checkpointing/00-uniform.md) | Store primal state every $k$-th step; on backward, re-run the Newton loop between checkpoints to rematerialize each step's factored Hessian; peak memory has a $T/k$-term for primal checkpoints plus a $k$-term for the segment's live activations, and compute is a fixed $\approx 2\times$ forward independent of $k$ (the trajectory replays exactly once). Lazy baseline, easy to implement |
| [Revolve algorithm](04-checkpointing/01-revolve.md) | [Griewank & Walther 2000](../appendices/00-references/02-adjoint.md#griewank-walther-2000)'s binomial checkpoint placement — $O(\log T)$ memory, $O(T \log T)$ compute with provably optimal constants for fixed checkpoint budgets; used by [adolC](https://github.com/coin-or/ADOL-C), [Dolfin-adjoint](https://www.dolfin-adjoint.org/), and JAX's `jax.checkpoint` at multi-level scheduling |
| [Memory-vs-compute tradeoff](04-checkpointing/02-tradeoff.md) | How `sim-soft`'s checkpoint budget scales with GPU VRAM (Phase E+) vs. host RAM (Phase D); when uniform is Pareto-dominant versus when Revolve's compute premium is worth paying; how contact active-set changes and adaptive-$\Delta t$ variance interact with both schedules |

Two claims Ch 04 rests on:

1. **Checkpointing is not optional at `sim-soft` scale.** At 30k DOFs the per-step tape footprint is dominated by the sparse Cholesky factor — order-of-hundreds-of-megabytes at the canonical-problem scene per [Ch 02 §02 memory](02-implicit-function/02-memory.md) — so the full uncheckpointed trajectory across a multi-hundred-to-thousand-step rollout runs into tens of gigabytes and approaches or exceeds typical workstation-GPU VRAM budgets. Uniform checkpointing at spacing $k$ stores primal only at $T/k$ checkpoints and replays one full forward pass on backward, giving peak memory with a $T/k$-sized primal stack plus a $k$-sized segment buffer of live activations, at a fixed $\approx 2\times$ forward-compute overhead independent of $k$. Revolve's binomial schedule cuts peak memory to $O(\log T) \cdot |\text{primal}|$ plus a bounded activation buffer at $O(T \log T)$ compute — for $T = 1000$ that is $\log_2 T \approx 10$ checkpoints live and $\approx 10 \times$ forward-compute overhead. For rollouts long enough that the uncheckpointed trajectory exceeds available memory, checkpointing is the only affordable path.

2. **The Phase D–E memory hierarchy is the real constraint.** Phase D's CPU-only implementation works against host RAM, which on workstation hardware is several times larger than GPU VRAM and carries no single-vs-double-precision ambiguity. Phase E's GPU port works against VRAM that is a factor of several tighter, with kernels often preferring single precision for throughput — both effects tilt the Phase E memory budget further than Phase D's. The plan is to ship uniform checkpointing in Phase D ([Build Order Phase D](../110-crate/03-build-order.md#the-committed-order)) as the simplest correct thing matching sim-soft's primal-much-smaller-than-factor asymmetry, and to re-evaluate in Phase E whether Revolve's $O(\log T)$ memory is worth its $O(T \log T)$ compute at the trajectory lengths Phase E workloads target. The [regression-vs-CPU gradcheck](../110-crate/04-testing/01-regression.md) guards the transition: whichever scheme Phase E ships must reproduce the Phase D CPU gradient to five digits.
