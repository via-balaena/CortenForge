# Hex8 — trilinear hexahedra

The [element-choice parent](../00-element-choice.md) made the rejection: `sim-soft` does not ship Hex8 because the upstream SDF → mesh workflow ([Part 7 Ch 01](../../../70-sdf-pipeline/01-tet-strategies.md)) does not have a robust hex-meshing answer. This leaf writes the Hex8 element down — its 8 nodes, its trilinear shape functions, the 2×2×2 Gauss quadrature it needs, the 24×24 per-element stiffness it produces — names what Hex8 is genuinely better at than Tet4 (so the rejection is not framed as an absolute), and identifies the upstream meshing problem that gates a future re-evaluation.

## Reference geometry and shape functions

A Hex8 element has 8 nodes at the corners of a brick-shaped element. Its shape functions on the reference cube $(\xi, \eta, \zeta) \in [-1, 1]^3$ are the trilinear products:

$$ N_i(\xi, \eta, \zeta) = \tfrac{1}{8}\,(1 + \xi_i \xi)(1 + \eta_i \eta)(1 + \zeta_i \zeta) $$

with $(\xi_i, \eta_i, \zeta_i) \in \{-1, +1\}^3$ the corner sign-tuple of node $i$. Each $N_i$ is 1 at its corner and 0 at the other 7. The deformation gradient $F = I + \sum_i u_i \otimes \nabla_X N_i$ is **trilinear in $(\xi, \eta, \zeta)$** — non-constant within the element, like Tet10 but with a different polynomial structure.

## Gauss-point quadrature and reduced-integration trade

Full integration uses a 2×2×2 = 8-point Gauss rule, exact for polynomial integrands of order ≤3 in each direction (the trilinear-shape-function stiffness integrand is order 2 per direction; 2×2×2 covers it with margin). Per-element constitutive call count is therefore 8 per Newton step under full integration — twice Tet10's 4.

**Reduced integration** drops the rule to 1 Gauss point at the centroid. This eliminates volumetric-locking issues that full integration suffers under near-incompressibility (Hex8 with 8-point integration locks similarly to Tet4 in the same regime), but introduces **hourglassing** — spurious zero-energy deformation modes the under-integrated stiffness matrix cannot resist. Production hex codes (LS-DYNA, Abaqus) ship reduced integration with hourglass-stabilization terms that penalize the spurious modes; the implementation complexity is non-trivial and a known recurring source of solver bugs in the FEM community. Selective reduced integration — full integration for the deviatoric part, 1-point for the volumetric part — is a third option that sidesteps both lock and hourglass, and is the typical industrial-FEM Hex8 default.

`sim-soft` makes none of these choices because the element is not shipped at all; the choices are listed here for the record so that a future Hex8 path can pick from the well-understood menu rather than re-deriving it.

## Per-element stiffness and DOF count

Hex8 has 24 DOFs per element (8 nodes × 3). The element stiffness matrix is $24 \times 24$, organized as an $8 \times 8$ block matrix of $3 \times 3$ blocks. The intra-element block sparsity is fully populated (all 64 blocks). Globally, the row-density per node depends on the mesh's hex-connectivity (regular hex grids have 8-element-per-vertex valence and thus low row-density; unstructured hex meshes from hex-dominant generators have higher and less uniform valence).

The [Part 8 Ch 01 BSR-3 layout](../../../80-gpu/01-sparse-matrix/01-bsr.md) would accept Hex8 directly — the $3 \times 3$ block size is keyed to vertex DOFs — so a future Hex8 path requires no GPU-storage change at the [Part 11 Ch 00 element module](../../../110-crate/00-module-layout/01-element.md) level beyond the new element-type's per-element assembly kernel.

## Where Hex8 actually wins

Hex8's strengths are real and not dismissed by `sim-soft`'s rejection of it as a primary element:

- **Per-DOF accuracy on aligned-grid problems.** When the deformation aligns with the hex grid axes (compression of a brick, bending of a rectangular cantilever), Hex8 reaches a target accuracy with fewer DOFs than Tet4 because the trilinear basis matches the deformation modes of axis-aligned problems naturally.
- **Less volumetric locking under near-incompressibility.** Selectively-reduced or B-bar Hex8 handles $\nu \to 0.5$ regimes more cleanly than Tet4 without the [mixed u-p machinery from Part 2 Ch 05](../../../20-materials/05-incompressibility/01-mixed-up.md). The same locking treatments apply to hex elements when needed but are usually not.
- **Industrial-FEM lineage.** Pressure vessels, structural steel, casting molds, anything blocky and amenable to extruded mesh layout — Hex8 is the right tool.

These are real strengths. They do not apply to `sim-soft`'s workload (compliant cavity-probe conformity on materials defined by SDFs with arbitrary topology), which is the workload-fit reason for the rejection, not a quality judgment on Hex8 itself.

## The upstream meshing problem

The technical gate is upstream of the element layer. Robust hexahedral meshing of an arbitrary input domain (and an SDF-defined domain in particular) is unsolved in the same sense that **robust tetrahedral meshing was unsolved before fTetWild**: heuristics worked on some inputs and failed on others, with no clean guarantees. fTetWild ([Hu et al. 2020](../../../appendices/00-references/03-diff-sim.md#hu-2020)) closed that gap for tets — any input mesh / SDF, robust output. The hex equivalent does not yet exist.

Hex-dominant generators (HexEx, Instant Meshes, polycube-based methods) produce hex meshes for some inputs and fall back to hex-dominant or fully tetrahedral output on others; production users typically iterate by hand on the hex-block layout. That manual hex-block-layout step is incompatible with the [Part 7 Ch 00 SDF-as-primitive commitment](../../../70-sdf-pipeline/00-sdf-primitive.md): the user is supposed to author an SDF in [cf-design](../../../70-sdf-pipeline/00-sdf-primitive/02-cf-design.md) and have a mesh appear, not author an SDF and then hand-place hex blocks across it. A Hex8 path would require the upstream meshing problem to be solved first, with the same robustness guarantee the tet path already has.

When the robust hex-meshing problem has a fTetWild-equivalent answer, `sim-soft` can revisit the rejection — adding Hex8 at the [`element/`](../../../110-crate/00-module-layout/01-element.md) module level is incremental, not architectural. Pass 1 plans no work toward this; the [Phase A–I roadmap](../../../120-roadmap/00-dependencies.md) does not have a Hex8 milestone.

## What this sub-leaf commits the book to

- **`sim-soft` does not ship Hex8.** Phase A through Phase I have no Hex8 element; the [`element/`](../../../110-crate/00-module-layout/01-element.md) module's element-type enumeration is `{Tet4, Tet10}`.
- **The rejection is workload-conditional, not absolute.** A future revision that adds robust SDF-to-hex meshing upstream can add Hex8 at the element layer without changing the [Part 11 Ch 01 trait surfaces](../../../110-crate/01-traits.md). The architectural commitment is to the trait surface, not to the element-type enumeration.
- **Reduced-integration and selective-integration design choices are documented but not chosen.** A future Hex8 path picks from this menu; `sim-soft` makes no commitment to a specific quadrature rule for an element it does not ship.
- **The gating problem is upstream, not in `sim-soft`.** Hex-from-SDF robustness is a meshing-research concern, not a soft-body-solver concern. The [Part 7 Ch 01 tet-strategies survey](../../../70-sdf-pipeline/01-tet-strategies.md) does not list a hex strategy because Pass 1 commits to the tet pipeline.
