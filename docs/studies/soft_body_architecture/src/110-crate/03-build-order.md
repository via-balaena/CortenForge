# Build order

This chapter names the order in which `sim-soft`'s ten modules ([Ch 00](00-module-layout.md)) get built. The order is constrained by the module-dependency graph — a module cannot be built before the modules it depends on — but the graph has several valid topological orderings and the choice among them matters. Different orderings trade off different things: time-to-first-working-solve versus time-to-first-differentiable-solve, CPU-first versus GPU-first, IPC-last versus IPC-integrated-early.

The order below is the one this book commits to. It is not the only defensible order; the rejected alternatives and the reasons for rejecting them are at the end of the chapter.

## The book is the spec

There is no Phase 0 of research, RFC-writing, or design-doc drafting preceding Phase A. That work IS Passes 1–3 of this mdbook. A consequence: Phase A cannot begin until the book is dense enough to determine module trait signatures unambiguously. The Pass-3 leaf chapters in Parts 2–9 are what make the [`material/`](00-module-layout/00-material.md), [`element/`](00-module-layout/01-element.md), [`contact/`](00-module-layout/03-contact.md), and [`solver/`](00-module-layout/04-solver.md) trait surfaces specific enough to compile against. The book stops being a design study and becomes a working specification at that point — Phase A's start condition is book-completion, not a separate RFC sign-off.

## The committed order

**Phase A — Standalone math.** `material/` and `mesh/` first. Both are pure and depend on nothing else in the crate. `material/` ships element-independent constitutive laws (neo-Hookean + linear as baseline + corotational as mid-point) with CPU evaluation and reverse-mode autograd via `sim-ml-chassis`'s existing CPU backend. `mesh/` ships tet mesh storage, adjacency, quality metrics, and Tet4-only initial fidelity. No elements, no solver, no contact. Each module has its own [gradcheck](04-testing/03-gradcheck.md) suite before anything that depends on it compiles. If `material/` or `mesh/` have bugs, they get caught in a 20-line test before they contaminate `solver/`.

**Phase B — Element + solver on one material.** `element/` (Tet4 first, Tet10 deferred to Phase H) composes `material/` with shape-function machinery. `solver/` lands a backward-Euler Newton loop on total potential energy with elastic-only terms — no contact — using a sparse direct solver on CPU: nalgebra for dense per-element matrices (3×3 deformation gradients, local stiffness, gradient accumulators), and [faer](https://github.com/sarah-quinones/faer-rs) for the global sparse factorization. faer is the pick because it is pure Rust (no FFI, no C/C++ dependency added to the build graph), because its factorizations are first-class objects that can be re-applied to arbitrary RHSes (exactly the shape Phase D's [IFT adjoint](../60-differentiability/02-implicit-function.md) consumes — one factor, many RHSes), and because sarah-ek's shipping cadence on sparse direct and iterative solvers lines up with our Phase B timing. The deliverable at the end of Phase B is a compressed cube of neo-Hookean material reaching static equilibrium under prescribed boundary conditions, with gradients of the equilibrium configuration w.r.t. material parameters passing a finite-difference gradcheck to 5 digits on a 100-tet mesh.

**Phase C — Contact.** `contact/` ships IPC barrier energy, CCD, smoothed Coulomb friction, and broad-phase BVH. Integration with `solver/` is the key deliverable: the Newton loop now optimizes elastic + contact + inertia jointly on the same total potential energy. At the end of Phase C, the [canonical problem](../10-physical/00-canonical.md) compiles and runs — a compliant cavity conforming to a rigid probe under squeeze, in equilibrium, on CPU, with contact pressure readout. Gradients through contact are exercised on a reduced-dimensional version of the problem where finite differences are tractable.

**Phase D — Readout + ml-chassis boundary.** `readout/` ships stress extraction, contact pressure, and the reward terms from [Part 1 Ch 01](../10-physical/01-reward.md) as smooth barriers. The [`sim-ml-chassis` coupling](02-coupling/03-ml-chassis.md) closes — `sim-soft` exposes the γ-locked `RewardBreakdown` per-term struct and a gradient in the shape [Part 10 Ch 00 — forward](../100-optimization/00-forward.md) consumes. At the end of Phase D, a synthetic 2-parameter design space optimizes end-to-end on CPU. **This is the "first working sim-soft" milestone** — every remaining phase improves something that already works.

Phase D is numerically complete but visually minimal — no shader preview, no rendered frame, no `sim-bevy` coupling. The [Part 1 Ch 03 thesis](../10-physical/03-thesis.md) commits to visual and physical being fused at the *final product* level; the build order reaches that state physics-first, with Phase I's visual layer completing the fusion. A Phase D preview would render against a Tet4 mesh with no subdivision and would exhibit the rim-deformation failure from [Part 1 Ch 02](../10-physical/02-what-goes-wrong.md) in its ugliest form. Showing an ugly "first working" demo is worse than showing no demo; Phase D's *working* is end-to-end-gradient, not end-to-end-beautiful.

**Phase E — GPU port of hot paths.** `gpu/` ports the solver's inner loop — sparse matrix-vector product, preconditioned CG, neo-Hookean per-element assembly, IPC barrier evaluation — to wgpu compute kernels. The autograd tape machinery itself lives in `sim-ml-chassis`'s GPU extension ([Part 8 Ch 04](../80-gpu/04-chassis-extension.md), keyed off the "own every line" commitment in [Part 6 Ch 00 Claim 3](../60-differentiability/00-what-autograd-needs.md)); Phase E's blocking prerequisite is that the chassis extension ships first, then `sim-soft`'s `gpu/` consumes it and registers its solver-specific VJPs against it. The forward solve runs on GPU; the autograd tape records GPU kernel calls and replays them in reverse for VJPs. Gradient correctness is regression-tested against the Phase D CPU path on every PR, on a small scene, to 5 digits. At the end of Phase E, the canonical problem runs at interactive rates: ≥ 30 FPS at an experience-mode resolution (~5k tets), ≥ 5 Hz at the design-mode resolution (~30k tets) on a consumer GPU. These targets are derived by scaling [NVIDIA Warp's](../00-context/02-sota/01-warp.md) published per-timestep neo-Hookean benchmark numbers to a 60/5 Hz frame budget on comparable tet counts — they are *targets*, not measurements of `sim-soft`, and will be revised up or down as Phase E benchmarks land.

**Phase F — Coupling boundaries.** `coupling/` closes [`sim-core`](02-coupling/00-mjcf.md) and [`sim-thermostat`](02-coupling/01-thermostat.md). The rigid probe now comes from `sim-core` rather than a hardcoded SDF; `sim-thermostat` provides the temperature field and receives dissipative heating. The co-simulation timestep coordination from [Part 5 Ch 03](../50-time-integration/03-coupling.md) goes live. At the end of Phase F, the canonical problem runs as a multi-sibling simulation exercising three crates.

**Phase G — SDF bridge.** `sdf_bridge/` ships the [SDF → tet pipeline](../70-sdf-pipeline/00-sdf-primitive.md). The `cf-design` coupling becomes live; geometry edits in `cf-design` trigger re-meshing and warm-started re-solve (per [Part 7 Ch 04](../70-sdf-pipeline/04-live-remesh.md)). At the end of Phase G, the design-mode loop supports live geometric authoring — the designer edits an SDF, the mesh re-derives, the solve warm-starts, and the reward updates.

**Phase H — Fidelity upgrades.** Tet10 elements, mixed u-p incompressibility (Mooney-Rivlin, Ogden), Prony viscoelasticity, HGO anisotropy, adaptive refinement driven by stress feedback. Each extends `material/` or `element/` without changing the trait surface from [Ch 01](01-traits.md), so they are additive. Each upgrade has its own gradcheck and [regression-vs-flex](04-testing/01-regression.md) test before landing.

**Phase I — Visual layer.** `sim-bevy` consumes per-vertex attributes from `readout/` to render subsurface scattering, contact-driven micro-wrinkles, thermal overlays. The physics stack does not change in Phase I; the deliverable is the visual layer from [Part 9 Ch 00 — SSS](../90-visual/00-sss.md) reaching the ceiling defined in [Part 0 Ch 01 — ceiling](../00-context/01-ceiling.md).

## Why this order

**Phase A before everything** because the two standalone modules compile in isolation. Gradcheck tests exist before any integration machinery — if `material/` or `mesh/` have a sign error, a 20-line test catches it rather than a three-module symptom.

**Element + solver before contact** because a solver that converges on contact-free problems is a hard prerequisite for debugging contact. A Newton loop that diverges on a cube in equilibrium tells us nothing about the IPC barrier, whereas a Newton loop that converges on a cube and then diverges the moment contact is added tells us exactly where the bug is. Contact is Phase C because it is the hardest single phase and deserves its own focus.

**CPU-first, GPU port in Phase E** because wgpu kernels are harder to debug than CPU reference implementations. Phase D ends with a working CPU solver; Phase E uses it as the oracle. Gradient correctness on GPU is regression-tested against CPU to 5 digits on every PR. Without that oracle, a GPU gradient bug would surface as "the optimizer is converging to something different" — weeks of debug time instead of a minute.

**Coupling boundaries after GPU** because the sibling couplings pull hardest on `sim-soft` when it is at its fastest and most complete. A slow CPU-only `sim-soft` co-simulated with `sim-core` spends all its time as the bottleneck; the coupling interface is harder to design under that constraint.

**SDF bridge after coupling** because the SDF → tet pipeline requires live re-meshing with warm-started state transfer. That assumes the solver already handles arbitrary tet meshes (Phase D) and can warm-start cleanly (Phase E). Doing the SDF bridge earlier would force re-meshing to handle solver cold-starts — a different and harder problem.

**Fidelity upgrades last** because each is additive and low-risk. Putting Tet10 in Phase B would force every subsequent phase to test against both Tet4 and Tet10. Putting viscoelasticity in Phase A would force every early gradcheck to span the time axis — again, a different problem than the static-equilibrium gradcheck Phase A actually wants to run.

**Visual layer last** because none of the physics phases need it. `sim-bevy` reads per-vertex attributes that already exist by Phase D (`readout/`); rendering them is a `sim-bevy` concern, not a `sim-soft` concern. Deferring the visual layer avoids the games-physics trap of letting rendering constraints shape the physics design.

## Alternatives rejected

**Contact first.** Landing IPC in Phase A has the advantage of front-loading the hardest module, but removes the oracle — a working elastic-only solver — that Phase C uses to debug IPC integration. Rejected.

**GPU-first.** Starting on wgpu avoids the CPU-to-GPU port in the middle; it pays for this by making every early module harder to debug. Rejected — the Phase D CPU solver is worth the eventual port cost as a regression baseline that outlives Phase E.

**Single-phase big-bang build.** Everything at once, test at the end. Rejected on principle. Incremental integration is cheaper than a big-bang final-week integration, and the ability to return to each phase's deliverable as a regression baseline is worth the phase structure on its own.

**Couple to `cf-design` before the SDF bridge.** `cf-design` can produce SDFs today; a direct coupling that bypasses `sdf_bridge/` and meshes the SDF once at construction would unblock early `cf-design` experiments. Rejected because it commits to a re-meshing path that Phase G has to undo. Coupling boundaries should close once, not twice.

**Visual layer in Phase B or C.** A working visual layer in Phase B would make the canonical problem look compelling earlier, which is motivationally appealing. Rejected because Phase B's mesh is Tet4-only and ugly, and because shader development against an unstable physics substrate produces throwaway shaders.
